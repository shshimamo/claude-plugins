# レビュールール更新スキル

## 事前準備

### 1. プロジェクト名の取得

以下の優先順位でプロジェクト名を取得する:

1. `git remote get-url origin` の出力からリポジトリ名を抽出
2. git管理外または失敗した場合は `basename $(pwd)` を使用

### 2. 既存ルールファイルの読み込み

以下の2ファイルを読み込む:

- `~/.review/common.md`
- `~/.review/<project-name>.md`

存在しない場合は新規作成する:

```
# Common Review Rules
```

```
# <project-name> Review Rules
```

## 処理フロー

### ステップ1: 直近のレビュー内容の確認

今の会話の中で `/review-custom` が実行されていた場合、その結果を参照する。
実行されていない場合は「直近のレビュー結果が見当たりません。先に `/review-custom` を実行してください。」と伝えて終了する。

### ステップ2: ルール候補の提案

レビュー結果から「今後のレビューで継続的にチェックすべきルール」を抽出して提案する。

判断基準:
- **common に追加すべき**: 言語・フレームワークを問わず普遍的なルール（セキュリティ、命名規則、エラーハンドリング等）
- **プロジェクト固有に追加すべき**: このプロジェクト特有のアーキテクチャ、規約、ドメインルール
- **追加不要**: 一度きりの指摘、すでに既存ルールに含まれる内容

以下の形式で提案する:

```
## ルール追加の提案

### ~/.review/common.md への追加候補
1. <ルール内容>
2. <ルール内容>

### ~/.review/<project-name>.md への追加候補
1. <ルール内容>
2. <ルール内容>

### 追加不要と判断したもの
- <内容>: <理由>

どれを追加しますか？（番号で指定、「common: 1,2 / project: 1」のように、またはすべて / なし）
```

### ステップ3: ユーザー確認

ユーザーの回答を受けて、指定された項目のみ追記する。

### ステップ4: ファイルへの追記

選択された項目を該当ファイルに追記する。

フォーマット（ファイル末尾に追記）:
```
## YYYY-MM-DD

- <ルール内容>
- <ルール内容>
```

追記後に「追加しました」と報告し、追記内容を表示する。
