# レビュールール更新スキル

## 事前準備

### 1. プロジェクト名の取得

以下の優先順位でプロジェクト名を取得する:

1. `git remote get-url origin` の出力からリポジトリ名を抽出
2. git管理外または失敗した場合は `basename $(pwd)` を使用

### 2. 既存ルールファイルの読み込み

以下の2ファイルを読み込む:

- `~/.claude-plugins/review/common.md`
- `~/.claude-plugins/review/<project-name>.md`

存在しない場合は新規作成する:

```
# Common Review Rules
```

```
# <project-name> Review Rules
```

## 処理フロー

### ステップ1: 会話内容の確認

今の会話全体を参照して、ルール化できる知見を探す。対象は以下を含む:

- `/review-custom` のレビュー結果
- 開発中に出てきたフィードバック・議論・気づき
- バグ修正の過程で判明したこと
- コードレビューのコメント

特定のコマンド実行に限らず、会話の中で「今後も意識すべき」と判断できる内容を広く拾う。
会話の中にルール化できる内容が見当たらない場合はその旨を伝えて終了する。

### ステップ2: ルール候補の提案

会話内容から「今後のレビューで継続的にチェックすべきルール」を抽出して提案する。

判断基準:
- **common に追加すべき**: 言語・フレームワークを問わず普遍的なルール（セキュリティ、命名規則、エラーハンドリング等）
- **プロジェクト固有に追加すべき**: このプロジェクト特有のアーキテクチャ、規約、ドメインルール
- **追加不要**: 一度きりの指摘、すでに既存ルールに含まれる内容

以下の形式で提案する:

```
## ルール追加の提案

### ~/.claude-plugins/review/common.md への追加候補
1. <ルール内容>
2. <ルール内容>

### ~/.claude-plugins/review/<project-name>.md への追加候補
1. <ルール内容>
2. <ルール内容>

### 追加不要と判断したもの
- <内容>: <理由>

どれを追加しますか？（番号で指定、「common: 1,2 / project: 1」のように、またはすべて / なし）
```

### ステップ3: ユーザー確認

ユーザーの回答を受けて、指定された項目のみ追記する。

### ステップ4: ファイルへの追記

選択された項目を該当ファイルに追記する。

フォーマット（ファイル末尾に追記）:
```
## YYYY-MM-DD

- <ルール内容>
- <ルール内容>
```

追記後に「追加しました」と報告し、追記内容を表示する。
