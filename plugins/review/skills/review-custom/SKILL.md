# カスタムコードレビュースキル

## 事前準備

### 1. CLAUDE.md / CLAUDE.local.md の読み込み

以下のファイルを読み込み、プロジェクトの規約・指示をレビューに反映する:

- `CLAUDE.md`（プロジェクトルートおよびカレントディレクトリ）
- `CLAUDE.local.md`（同上）

存在しない場合はスキップする。

### 3. プロジェクト名の取得

以下の優先順位でプロジェクト名を取得する:

1. `git remote get-url origin` の出力からリポジトリ名を抽出（例: `github.com/user/my-repo` → `my-repo`）
2. git管理外または失敗した場合は `basename $(pwd)` を使用

### 4. 追加ルールファイルの読み込み

以下のファイルを読み込む:

- `~/.review/common.md` — 全プロジェクト共通のレビュー観点（存在しない場合は `# Common Review Rules` の内容で新規作成）
- `~/.review/<project-name>.md` — このプロジェクト固有のレビュー観点（存在しない場合は `# <project-name> Review Rules` の内容で新規作成）
- `.claude/skill-evolve/review-patterns.md` — skill-evolve が週次自動生成するパターン（存在しない場合はスキップ）

### 5. 変更内容の取得

`git diff HEAD` を実行して変更内容を取得する。
ステージング済みの変更がある場合は `git diff --cached` も確認する。
変更がない場合はユーザーに伝えて終了する。

## レビュー実行

以下の観点でコードレビューを行う。

### 標準レビュー観点（公式 /review 相当）

- **バグ・ロジックエラー**: 明らかなバグ、エッジケースの見落とし、条件分岐の誤り
- **セキュリティ**: 入力検証、認証・認可、機密情報の扱い、インジェクション脆弱性
- **パフォーマンス**: 不必要なループ、N+1クエリ、メモリリーク
- **可読性・保守性**: 命名の明確さ、複雑すぎる処理、重複コード
- **エラーハンドリング**: 例外処理の漏れ、エラーメッセージの適切さ
- **テスト**: テストの欠如、テストケースの不足

### 追加レビュー観点

読み込んだ `common.md`・`<project-name>.md`・`review-patterns.md` の内容をレビュー観点として加える。
ファイルが空または存在しない場合はスキップする。

## 出力フォーマット

```
## コードレビュー結果

**対象**: <変更ファイル一覧>
**プロジェクト固有ルール**: ~/.review/<project-name>.md
**共通ルール**: ~/.review/common.md

---

### 重大な問題 🔴
（あれば記載、なければ省略）

### 軽微な問題 / 改善提案 🟡
（あれば記載、なければ省略）

### 良い点 ✅
（あれば記載、なければ省略）

---

**総評**: <全体的なコメント>

> レビューが完了しました。今回の結果を踏まえてルールに追加したい項目があれば `/review-update` を実行してください。
```

各指摘には以下を含める:
- 該当ファイル名と行番号（わかる場合）
- 問題の説明
- 改善案（コードスニペットを含めると良い）
