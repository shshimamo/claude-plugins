# Repo Know スキル

保存先: `~/.repo-know/<project-name>/`
知識ファイル:
- `~/.repo-know/<project-name>/tech.md` — 技術スタック・アーキテクチャ・ライブラリ
- `~/.repo-know/<project-name>/domain.md` — ドメイン知識・用語・ビジネスルール
- `~/.repo-know/<project-name>/decisions.md` — 設計判断・採用した方針とその理由

## 知識ファイルのフォーマット

~~~markdown
# Tech Knowledge - <project-name>

## YYYY-MM-DD

- <知識>
- <知識>

## YYYY-MM-DD

- <知識>
~~~

---

## 呼び出しパターン

### 引数なし（`/repo-know`）

**ステップ1: プロジェクト名の取得**

以下の優先順位で取得する:
1. `git remote get-url origin` からリポジトリ名を抽出（例: `github.com/user/my-repo` → `my-repo`）
2. git管理外または失敗した場合は `basename $(pwd)` を使用

**ステップ2: 既存知識の読み込み**

`~/.repo-know/<project-name>/` の各ファイルを読み込む。存在しない場合は空として扱う。

**ステップ3: 会話からの知識抽出**

現在の会話全体を振り返り、以下の観点でプロジェクトに関する知識を抽出する:

- **tech**: 使用技術・フレームワーク・ライブラリ、アーキテクチャ、インフラ、ファイル構成、開発フロー
- **domain**: ビジネスルール、ドメイン用語、エンティティの意味・関係、ユースケース
- **decisions**: 採用した設計方針・その理由、却下した選択肢とその理由、トレードオフ

すでに既存ファイルに記載されている内容は候補から除外する。
新しい知識・補足・修正のみを候補として挙げる。

**ステップ4: 候補の提案**

以下の形式でユーザーに提示する:

~~~
## 知識の抽出結果（<project-name>）

### tech.md への追加候補
1. GoとGin frameworkを使用したREST API
2. PostgreSQLをメインDB、Redisをキャッシュに使用

### domain.md への追加候補
1. 「注文」はカートから確定した状態を指す（未確定はカート）
2. ユーザーは「顧客」と「管理者」の2種類

### decisions.md への追加候補
1. 認証にJWTを採用（セッション管理のコストを避けるため）

### 追加不要と判断したもの
- GoのバージョンはREADMEに記載済みのため除外

どれを保存しますか？（番号で指定、例: "tech: 1,2 / domain: 1 / decisions: 1"、またはすべて / なし）
~~~

抽出できる知識がない場合は「この会話から新しい知識は見つかりませんでした」と伝えて終了する。

**ステップ5: ユーザー確認**

ユーザーの選択を受けて、指定された項目のみ保存する。

**ステップ6: ファイルへの追記**

`~/.repo-know/<project-name>/` ディレクトリを作成し、選択された項目を該当ファイルに追記する。

追記後に保存内容を表示して完了を報告する。

---

### `show`

**ステップ1: プロジェクト名の取得**

引数なしの場合と同様にプロジェクト名を取得する。

**ステップ2: 知識ファイルの読み込み・表示**

`~/.repo-know/<project-name>/` の全ファイルを読み込んで表示する。

ファイルが存在しない・空の場合は「まだ知識が蓄積されていません。`/repo-know` で保存してください。」と伝える。

表示フォーマット:

~~~
## <project-name> の知識

### 技術スタック・アーキテクチャ（tech.md）
（内容）

### ドメイン知識（domain.md）
（内容）

### 設計判断（decisions.md）
（内容）
~~~
