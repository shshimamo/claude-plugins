# Todo管理スキル

Todoファイル: `~/.todo/todos.md`

まず必ずこのファイルを読んでから処理する。

## 呼び出しパターン

### 引数なし（`/todo`）
1. 未完了タスクの一覧を表示（子タスクはインデントして表示）
2. 完了タスク数を簡潔に表示
3. 今やるべきタスクを提案（登録順を基本に）

### `add <タスク名>`
1. ファイルを読んで既存の未完了タスク一覧を把握する
2. 子タスク判定を行う（下記「子タスク判定ロジック」を参照）
3. 子タスクにすべきと判断した場合はユーザーに確認してから登録
4. 子タスクにしない場合はトップレベルタスクとして登録

### `add <タスク名> > <親タスク名>`（明示的な親指定）
- 指定した親タスクの子として登録する
- 親タスクが存在しない場合はエラーを伝える
- 確認は不要、そのまま登録する

### `done <タスク名>`
- 未完了セクションから該当タスクを探す（部分一致でOK、子タスクも対象）
- 完了セクションに移動
- フォーマット: `- [x] <タスク名> (登録: YYYY-MM-DD HH:MM, 完了: YYYY-MM-DD HH:MM)`
- 登録時刻はそのまま保持し、完了時刻を追記
- 子タスクを完了させた場合、未完了の兄弟タスクがあれば次の子タスクを提案する
- 親タスクの完了は手動（自動完了しない）

### `rm <タスク名>`
- 未完了セクションから該当タスクを探す（部分一致でOK）
- 子タスクも含めてゴミ箱セクションに移動する
- フォーマットはそのまま保持（`[ ]` のまま移動）

## 子タスク判定ロジック

`add <タスク名>` で親指定がない場合、以下の基準で判定する。

**子タスクと判断するケース：**
- 新タスクが既存タスクの具体的な実装ステップ・サブ作業に見える
- 新タスクの内容が既存タスクのスコープに明確に含まれる
- 例：「ECサイト開発」があるときに「ログイン機能実装」を登録 → 子タスク候補

**トップレベルと判断するケース：**
- 既存タスクと独立した別の目的・テーマを持つ
- 関連はあっても並列関係（同レベルの作業）
- 既存タスクが1件以下しかない

**判定後の動作：**
- 子タスク候補が1件見つかった場合：「`<親タスク名>` の子タスクとして登録しますか？」と確認
- 候補が複数ある場合：候補をリストアップして選ばせる（「どの親タスクの子にしますか？それともトップレベル？」）
- 子タスクにしない場合：トップレベルに登録

## ファイルフォーマット

```markdown
# Todo

## 未完了
- [ ] タスク名 (登録: 2026-05-17 10:00)
  - [ ] 子タスク名 (登録: 2026-05-17 11:00)
  - [ ] 子タスク名2 (登録: 2026-05-17 11:30)
- [ ] 別タスク (登録: 2026-05-17 12:00)

## 完了
- [x] タスク名 (登録: 2026-05-17 09:00, 完了: 2026-05-17 12:00)
  - [x] 子タスク名 (登録: 2026-05-17 09:30, 完了: 2026-05-17 11:00)

## ゴミ箱
- [ ] 削除したタスク (登録: 2026-05-17 08:00)
  - [ ] 削除した子タスク (登録: 2026-05-17 08:30)
```

- 子タスクは親タスクの直下に2スペースインデントで記述
- 子タスクの完了時も親タスクの直下に残す（完了セクションへの移動は親タスク単位ではなく個別）
- ファイルが存在しない場合は上記フォーマットで新規作成する
