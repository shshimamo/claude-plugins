# Dev スキル

保存先: `~/.dev/<project-name>/<feature-name>/`

```
~/.dev/<project>/
└── <feature-name>/
    ├── investigation/
    │   ├── summary.md       # 全体サマリー・影響範囲
    │   ├── architecture.md  # アーキテクチャの理解
    │   └── <topic>.md       # テーマ別（auth, db, api など）
    ├── plan.md              # 実装計画・方針
    ├── tasks.md             # タスクリスト（進捗管理）
    └── artifacts/
        ├── sequence.md          # シーケンス図（Mermaid）
        ├── db.md                # DB設計（ERD）
        ├── api-spec.md          # API仕様（エンドポイント・リクエスト/レスポンス）
        ├── components.md        # コンポーネント構成・責務（複雑なシステムの場合）
        ├── testing-strategy.md  # テスト戦略・カバレッジ方針（任意）
        └── nfr.md               # 非機能要件（性能・セキュリティ）（任意）
```

## 呼び出しパターン

### `plan`

#### ステップ1: 仕様ヒアリング

以下を順番に質問して情報を収集する。回答に応じて深掘りしてよい。

**質問1: 仕様**
```
どんな機能を実装したいですか？
```

**質問2: 関連リポジトリ**
```
関連するリポジトリはありますか？（複数ある場合はすべて教えてください）
```

関連リポジトリがある場合はそのパスまたは名前を記録する。ない場合はカレントリポジトリのみ対象。

**質問3: 参考文献**
```
参考にしたいドキュメント・URL・資料はありますか？（仕様書、設計書、記事など）
```

参考文献がある場合は内容を読み込み、調査・計画フェーズで活用する。ない場合はスキップ。

#### ステップ2: プロジェクト名・フィーチャー名の取得

`git remote get-url origin`（失敗した場合は `basename $(pwd)`）でプロジェクト名を自動取得し、ユーザーに確認する:

> 対象リポジトリ: `<自動取得したプロジェクト名>`
> このリポジトリで合っていますか？（違う場合は正しい名前を教えてください）

確定したプロジェクト名を以降のすべての操作（作業ログの保存先・repo-know の読み書き）で使う。

フィーチャー名: 仕様から短い名前を生成してユーザーに確認（例: `user-auth`, `order-refund`）。
保存先ディレクトリ名になるため英数字・ハイフン推奨。

プロジェクト名が確定したら `~/.repo-know/<project-name>/` の各ファイルを読み込む（存在しない場合はスキップ）。
読み込んだ内容を既存コード調査・計画フェーズのコンテキストとして活用する。

#### ステップ3: タスク種別の判断

仕様をもとに以下から判断してユーザーに確認する:

- **new**: 新機能・新規モジュールの開発
- **feature**: 既存機能への追加・拡張
- **bugfix**: バグ修正
- **refactor**: リファクタリング

#### ステップ4: 既存コードの調査

調査の目的は「仕様を実装するために必要な既存コードの理解」。単なるファイル検索ではなく、コードの文脈と構造を理解すること。

関連リポジトリが複数ある場合は、それぞれに対して以下の手順を実施する。

**並列サブエージェントで調査する:**

以下の観点を Agent ツールで並列調査し、効率的に全体像を把握する:

- **コードベース構造**: ディレクトリ構成・主要ファイル・アーキテクチャパターンの把握
- **関連ファイル検索**: 仕様のキーワード（クラス名・関数名・用語・エンドポイント）で Grep し、関連ファイルを特定
- **既存実装パターン**: 類似機能の実装を調査し、コーディング規約・パターンを把握
- **参照追跡**: 特定したファイルを Read して import・呼び出し・型定義・DBスキーマ・テストファイルを芋づる式に追跡

各エージェントの調査結果を統合し、仕様の実装に必要な全体像が把握できるまで追加調査を行う。

**調査しながら随時 `investigation/` に記録する:**

調査内容に応じてファイルを作成・追記する。最低限 `summary.md` は必ず作成する。

`investigation/summary.md`:
~~~markdown
# Investigation Summary - <feature-name>

## 仕様の理解
（仕様を自分の言葉で整理したもの）

## 関連ファイル
- `path/to/file.go`: （このファイルの役割）
- `path/to/model.go`: （このファイルの役割）

## 影響範囲
（変更が必要なファイルと理由）

## 不明点・リスク

| # | 内容 | 影響 | 状態 |
|---|------|------|------|
| 1 | （不明点・リスク） | 高/中/低 | 未解決/解決済 |
~~~

`investigation/architecture.md`（アーキテクチャが複雑な場合）:
~~~markdown
# Architecture - <feature-name>

（コードを読んで理解したアーキテクチャ・レイヤー構造・パターン）
~~~

`investigation/<topic>.md`（テーマごとに必要に応じて作成）:
- `auth.md`: 認証・認可の仕組み
- `db.md`: DB構造・クエリパターン
- `api.md`: APIエンドポイント・リクエスト/レスポンス形式
- その他調査で発見したテーマに応じて自由に追加

#### ステップ4.5: repo-know への知識保存

調査で得られたプロジェクト固有の知識を `~/.repo-know/<project-name>/` に保存する。

調査結果を振り返り、以下の観点で知識を抽出する:
- **tech**: 調査で判明した技術スタック・アーキテクチャ・実装パターン
- **domain**: 判明したビジネスルール・ドメイン用語・エンティティの意味

既存ファイルと重複しないもののみ候補として提示し、ユーザー確認後に追記する。
候補がない場合はスキップ。

#### ステップ5: 計画の精緻化

仕様 + 調査結果をもとに実装計画を作成してユーザーと対話する。
不明点や選択肢がある場合はユーザーに確認しながら詰める。

`plan.md` を作成する:

~~~markdown
# Plan - <feature-name>

## 概要
（何を、なぜ実装するか）

## 実装方針
（どのようにアプローチするか）

## 影響範囲
（変更するファイル・モジュール一覧）

## 考慮事項
（パフォーマンス・セキュリティ・後方互換性など）

## トレーサビリティ

| 要件 | コンポーネント | API | データ |
|------|--------------|-----|--------|
| （要件名） | （担当コンポーネント） | （エンドポイント） | （エンティティ） |
~~~

#### ステップ6: アウトプット生成

タスク種別に応じて以下のアーティファクトを生成する:

| 種別 | 生成物 |
|------|-------|
| new | シーケンス図・DB設計・API仕様・コンポーネント構成・タスクリスト |
| feature | シーケンス図・API仕様（該当する場合）・タスクリスト |
| bugfix | 原因分析・修正方針・タスクリスト |
| refactor | 変更範囲・タスクリスト |

テスト戦略（`testing-strategy.md`）・非機能要件（`nfr.md`）は必要に応じて生成する。

**シーケンス図** (`artifacts/sequence.md`): Mermaid形式で記述

~~~markdown
# Sequence - <feature-name>

```mermaid
sequenceDiagram
    ...
```
~~~

**DB設計** (`artifacts/db.md`): 追加・変更するテーブル・カラムをMermaid ERD形式で記述

**API仕様** (`artifacts/api-spec.md`): エンドポイント・リクエスト/レスポンス仕様

~~~markdown
# API Spec - <feature-name>

**Base Path**: `/api/xxx`

## POST /api/xxx
- **概要**: ...
- **リクエスト**: `{ "field": "type" }`
- **レスポンス 201**: `{ "id": "...", ... }`
- **エラー**: 400, 404, ...
~~~

**コンポーネント構成** (`artifacts/components.md`): コンポーネントの責務・依存関係（複雑なシステムの場合）

**テスト戦略** (`artifacts/testing-strategy.md`): テスト種別・フレームワーク・カバレッジ方針（任意）

**非機能要件** (`artifacts/nfr.md`): 性能目標・セキュリティ要件・スケーラビリティ（任意）

**タスクリスト** (`tasks.md`):

~~~markdown
# Tasks - <feature-name>

## 基本情報
- ブランチ: `feature/xxx`
- チケット: #123（任意）
- 開始日: YYYY-MM-DD

## 要件サマリー
- **背景**: なぜこの機能が必要か
- **受入条件**: 完了の定義
- **スコープ外**: やらないこと

---

## タスク

| # | タスク名 | 状態 | 対象ファイル | 実装詳細 | 依存 |
|---|---------|------|------------|---------|------|
| 1 | xxx | 未着手 | `path/to/file` | 何をするか | - |
| 2 | yyy | 未着手 | `path/to/file` | 何をするか | #1 |

**進捗**: 0 / 2 完了

---

## 作業再開ガイド
- **最後のタスク**: -
- **次のアクション**: -
- **メモ**: -

---

## 作業ログ
- YYYY-MM-DD: （実施内容・判断・保留事項）
~~~

#### ステップ7: ブラッシュアップ

生成したアウトプットをユーザーに提示してフィードバックを求める。
ユーザーの指摘を反映して各ファイルを更新する。
「実装を開始してよいか」の確認を取る。

---

### `exec`

#### ステップ1: 計画の読み込み

プロジェクト名を取得後、`~/.dev/<project>/` 配下のフィーチャーディレクトリ一覧を表示してユーザーに選択させる。
選択されたフィーチャーの `tasks.md`・`plan.md`・`investigation/` のファイルを読み込む。

#### ステップ2: 進捗確認・アクション選択

`tasks.md` の状態を確認し、現在の進捗とアクションメニューを表示する:

~~~
## 進捗状況 - <feature-name>

進捗: N / M 完了

作業再開ガイド:
  最後のタスク: ...
  次のアクション: ...
  メモ: ...

### 次のタスク
| # | タスク名 | 対象ファイル | 実装詳細 | 依存 |
|---|---------|------------|---------|------|
| N | ...     | ...        | ...     | ...  |

### アクションを選んでください
1. 次のタスクを開始
2. 特定のタスクを選んで実行（番号を指定）
3. 並列開発モード（依存関係のないタスクをサブエージェントで並列実行）
4. タスクを完了済みにする（番号を指定）
5. 課題・メモを記録
~~~

#### ステップ3: タスクの実行

ユーザーの選択に応じて実行する。
実装時は `plan.md` と `investigation/` のファイルを参照して一貫性を保つ。

**並列開発モード（アクション3）— Agent Teams を使用:**

1. **wave 分析**: tasks.md の「依存」列を読んで wave に分割する
   ```
   wave 1: 依存なし（-）のタスク → 並列実行可能
   wave 2: wave 1 のタスクに依存するタスク → wave 1 完了後に実行
   wave 3: wave 2 依存 → ...
   ```
2. **チーム作成**: `TeamCreate` でフィーチャー名のチームを作成
3. **wave 1 実行**: 依存なしタスクごとに `Agent` ツールでチームメンバーを起動し `TaskUpdate` で割り当て
4. **完了待ち**: メンバーからの完了報告を受け取る
5. **次の wave**: アンブロックされたタスクを新しいメンバーに割り当て、wave が終わるまで繰り返す
6. **統合・検証**: 全タスク完了後、変更内容を統合しコンフリクト・不整合を解消
7. **チーム解散**: `TeamDelete` でチームを削除

#### ステップ4: tasks.md の更新

タスク完了後、`tasks.md` のタスクテーブルで該当行の状態を「完了」に更新し、進捗カウントを更新する。
「作業再開ガイド」と「作業ログ」を現在の状態に更新する。

アクションメニューに戻る。
