# Dev スキル

保存先: `~/.dev/<project-name>/<feature-name>/`

```
~/.dev/<project>/
└── <feature-name>/
    ├── investigation/
    │   ├── summary.md       # 全体サマリー・影響範囲
    │   ├── architecture.md  # アーキテクチャの理解
    │   └── <topic>.md       # テーマ別（auth, db, api など）
    ├── plan.md              # 実装計画・方針
    ├── tasks.md             # タスクリスト（進捗管理）
    └── artifacts/
        ├── sequence.md      # シーケンス図（Mermaid）
        └── db.md            # DB設計
```

## 呼び出しパターン

### `plan`

#### ステップ1: 仕様ヒアリング

以下を順番に質問して情報を収集する。回答に応じて深掘りしてよい。

**質問1: 仕様**
```
どんな機能を実装したいですか？
```

**質問2: 関連リポジトリ**
```
関連するリポジトリはありますか？（複数ある場合はすべて教えてください）
```

関連リポジトリがある場合はそのパスまたは名前を記録する。ない場合はカレントリポジトリのみ対象。

**質問3: 参考文献**
```
参考にしたいドキュメント・URL・資料はありますか？（仕様書、設計書、記事など）
```

参考文献がある場合は内容を読み込み、調査・計画フェーズで活用する。ない場合はスキップ。

#### ステップ2: プロジェクト名・フィーチャー名の取得

プロジェクト名: `git remote get-url origin` からリポジトリ名を抽出。失敗した場合は `basename $(pwd)`。

フィーチャー名: 仕様から短い名前を生成してユーザーに確認（例: `user-auth`, `order-refund`）。
保存先ディレクトリ名になるため英数字・ハイフン推奨。

#### ステップ3: タスク種別の判断

仕様をもとに以下から判断してユーザーに確認する:

- **new**: 新機能・新規モジュールの開発
- **feature**: 既存機能への追加・拡張
- **bugfix**: バグ修正
- **refactor**: リファクタリング

#### ステップ4: 既存コードの調査

調査の目的は「仕様を実装するために必要な既存コードの理解」。単なるファイル検索ではなく、コードの文脈と構造を理解すること。

関連リポジトリが複数ある場合は、それぞれに対して以下の手順を実施する。

**調査手順:**

1. **キーワード抽出**: 仕様から関連するクラス名・関数名・用語・エンドポイントを抽出
2. **初期検索**: Grep でキーワードに関連するファイルを特定
3. **参照追跡（芋づる式）**: 特定したファイルを Read して以下を把握し、さらに追跡する:
   - import/require している外部モジュール・内部ファイル
   - 呼び出している関数・メソッド・クラス
   - 影響を受けるインターフェース・型定義
   - 関連するDBスキーマ・マイグレーションファイル
   - テストファイル
4. **理解が深まるまで繰り返す**: 仕様の実装に必要な全体像が把握できるまで追跡を続ける

**調査しながら随時 `investigation/` に記録する:**

調査内容に応じてファイルを作成・追記する。最低限 `summary.md` は必ず作成する。

`investigation/summary.md`:
~~~markdown
# Investigation Summary - <feature-name>

## 仕様の理解
（仕様を自分の言葉で整理したもの）

## 関連ファイル
- `path/to/file.go`: （このファイルの役割）
- `path/to/model.go`: （このファイルの役割）

## 影響範囲
（変更が必要なファイルと理由）

## 不明点・リスク
（実装時に注意すべき点、不確かな箇所）
~~~

`investigation/architecture.md`（アーキテクチャが複雑な場合）:
~~~markdown
# Architecture - <feature-name>

（コードを読んで理解したアーキテクチャ・レイヤー構造・パターン）
~~~

`investigation/<topic>.md`（テーマごとに必要に応じて作成）:
- `auth.md`: 認証・認可の仕組み
- `db.md`: DB構造・クエリパターン
- `api.md`: APIエンドポイント・リクエスト/レスポンス形式
- その他調査で発見したテーマに応じて自由に追加

#### ステップ5: 計画の精緻化

仕様 + 調査結果をもとに実装計画を作成してユーザーと対話する。
不明点や選択肢がある場合はユーザーに確認しながら詰める。

`plan.md` を作成する:

~~~markdown
# Plan - <feature-name>

## 概要
（何を、なぜ実装するか）

## 実装方針
（どのようにアプローチするか）

## 影響範囲
（変更するファイル・モジュール一覧）

## 考慮事項
（パフォーマンス・セキュリティ・後方互換性など）
~~~

#### ステップ6: アウトプット生成

タスク種別に応じて以下のアーティファクトを生成する:

| 種別 | 生成物 |
|------|-------|
| new | シーケンス図・DB設計・タスクリスト |
| feature | シーケンス図・タスクリスト |
| bugfix | 原因分析・修正方針・タスクリスト |
| refactor | 変更範囲・タスクリスト |

**シーケンス図** (`artifacts/sequence.md`): Mermaid形式で記述

~~~markdown
# Sequence - <feature-name>

```mermaid
sequenceDiagram
    ...
```
~~~

**DB設計** (`artifacts/db.md`): 追加・変更するテーブル・カラムをMermaid ERD形式で記述

**タスクリスト** (`tasks.md`):

~~~markdown
# Tasks - <feature-name>

## 基本情報
- ブランチ: `feature/xxx`
- チケット: #123（任意）
- 開始日: YYYY-MM-DD

## 要件サマリー
- **背景**: なぜこの機能が必要か
- **受入条件**: 完了の定義
- **スコープ外**: やらないこと

---

## タスク

| # | タスク名 | 状態 | 対象ファイル | 実装詳細 |
|---|---------|------|------------|---------|
| 1 | xxx | 未着手 | `path/to/file` | 何をするか |
| 2 | yyy | 未着手 | `path/to/file` | 何をするか |

**進捗**: 0 / 2 完了

---

## 作業再開ガイド
- **最後のタスク**: -
- **次のアクション**: -
- **メモ**: -
~~~

#### ステップ7: ブラッシュアップ

生成したアウトプットをユーザーに提示してフィードバックを求める。
ユーザーの指摘を反映して各ファイルを更新する。
「実装を開始してよいか」の確認を取る。

---

### `exec`

#### ステップ1: 計画の読み込み

プロジェクト名を取得後、`~/.dev/<project>/` 配下のフィーチャーディレクトリ一覧を表示してユーザーに選択させる。
選択されたフィーチャーの `tasks.md` と `plan.md` を読み込む。

#### ステップ2: 進捗確認

`tasks.md` の状態を確認し、現在の進捗を表示する:

~~~
## 進捗状況 - <feature-name>

進捗: N / M 完了

作業再開ガイド:
  最後のタスク: ...
  次のアクション: ...
  メモ: ...

### 次のタスク
| # | タスク名 | 対象ファイル | 実装詳細 |
|---|---------|------------|---------|
| N | ...     | ...        | ...     |

実装を開始しますか？
~~~

#### ステップ3: タスクの実行

ユーザーの確認後、タスクを実装する。
実装時は `plan.md` と `investigation/` のファイルを参照して一貫性を保つ。

#### ステップ4: tasks.md の更新

タスク完了後、`tasks.md` のタスクテーブルで該当行の状態を「完了」に更新し、進捗カウントを更新する。
「作業再開ガイド」を現在の状態に更新する。

次のタスクに進むか確認する。
