# Plan - dev-review-cmd

## 概要

devスキルに `/dev review` コマンドを追加する。
`/dev exec` で実装したコードをAIがレビューし、問題点・改善提案を重大度別に整理して出力するコマンド。
開発者がセルフレビューの代わりにAIへ投げることで、コードレビューの品質担保と手戻りを削減する。

## 実装方針

`SKILL.md` に `review` セクションを追記する。既存の `plan` / `exec` セクションと同構造（ステップ番号付き）で記述し、後方互換性を維持する。

### reviewコマンドのフロー

1. **対象の特定**: `git diff HEAD` でレビュー対象を取得。ユーザーがブランチ・コミット範囲を任意指定可能。git管理外の場合はエラーメッセージで案内して終了。
2. **コンテキスト収集**: プロジェクト名・フィーチャー名を確認し、`~/.dev/<project>/<feature>/plan.md` と `tasks.md` が存在すれば読み込んで設計意図をレビューに活用。存在しない場合はスキップ。
3. **レビュー実行**: 取得コードを以下の観点でレビュー:
   - 正確性（バグ・ロジックエラー・エッジケース）
   - セキュリティ（インジェクション・認証漏れ・機密情報の扱い）
   - パフォーマンス（N+1・不要な処理・メモリリーク）
   - 可読性・保守性（命名・複雑度・重複コード）
   - エラーハンドリング（例外処理の漏れ）
   - テスト漏れ
4. **結果出力**: 重大度別（Critical / Warning / Suggestion）に整理してユーザーに提示し、`~/.dev/<project>/<feature>/review.md` に保存。

### review.md のテンプレート

```markdown
# Review - <feature-name>

**日時**: YYYY-MM-DD
**対象**: <git diff の対象ブランチ/コミット範囲>
**変更ファイル**: <ファイル一覧>

---

## Critical（要修正）
<!-- バグ・セキュリティ問題など即対応が必要なもの -->

## Warning（推奨修正）
<!-- パフォーマンス問題・エラーハンドリング漏れなど -->

## Suggestion（改善提案）
<!-- 可読性・保守性の改善提案 -->

## 良い点
<!-- 適切な実装・工夫を評価 -->

---

**総評**: <全体的なコメント>
```

| `plugins/dev/skills/dev/SKILL.md` | `review` コマンドセクション追記、フロントマターのdescription更新 |

## 考慮事項

- **後方互換性**: 既存の `plan` / `exec` セクションは変更しない
- **git diff 依存**: git管理外のプロジェクトでは動作しない（エラーメッセージで案内）
- **大きな差分**: 差分が大きい場合はファイル単位での分割レビューを推奨する旨をSKILL.mdに明示
- **フィーチャー未紐付けの場合**: `plan.md` がなくても単体で動作可能にする
- **保存先の柔軟性**: フィーチャー名が不明な場合は `~/.dev/<project>/review-<date>.md` に保存
