# Investigate Bug スキル

保存先: `~/.investigate/<project-name>/<investigation-name>/`

```
~/.investigate/<project>/
└── <investigation-name>/
    ├── investigation/
    │   └── summary.md        # 調査サマリー・関連ファイル・リスク
    ├── hypothesis_log.md      # 調査した仮説と棄却理由
    ├── root_cause.md          # 根本原因
    └── fix_proposal.md        # 修正案
```

---

## ガードレール

調査の信頼性を保つために以下を必ず守る:

- **推測禁止**: コードを読んで確認した事実のみを記録する。未検証の内容は「仮説:」と明記する
- **パス記録**: 調査した経路・確認したファイル・棄却した仮説は必ず記録する（同じ道を二度辿らないため）
- **3回失敗で停止**: 同じ方向の仮説が3回連続で否定されたらユーザーに報告して指示を待つ

---

## ステップ1: ヒアリング

以下を順番に質問する。

**質問1: エラー情報**
```
エラーログ・スタックトレース・Issueの内容を教えてください。
```

**質問2: 発生状況**
```
いつ・どんな操作で発生しますか？再現手順があれば教えてください。
```

**質問3: 関連リポジトリ**
```
関連するリポジトリはありますか？（複数ある場合はすべて教えてください）
```

**質問4: 関連リンク**
```
関連する URL・チケット・ダッシュボードはありますか？（Issue、Confluenceページ、Sentryイベント、Grafanaダッシュボード、Asanaタスクなど）
```

リンクが提供された場合は以下の方針でアクセスする:

| サービス | アクセス方法 |
|---------|------------|
| GitHub（github.com） | `gh` コマンドで Issue / PR / コードを取得 |
| Confluence | MCP を使用して検索・取得 |
| Asana | MCP を使用してタスク情報を取得 |
| Grafana | MCP を使用してダッシュボード・クエリを取得 |
| Sentry | MCP を使用してエラー詳細・スタックトレースを取得 |
| その他 | 対応する MCP があれば使用。なければユーザーにアクセス方法を確認 |

## ステップ2: プロジェクト名・調査名の取得

プロジェクト名: `git remote get-url origin` からリポジトリ名を抽出。失敗した場合は `basename $(pwd)`。

調査名: バグの内容から短い名前を生成してユーザーに確認（例: `nil-pointer-user-auth`, `timeout-payment-api`）。英数字・ハイフン推奨。

## ステップ3: 並列調査

Agent ツールで以下を並列調査する:

- **エラー箇所の特定**: スタックトレース・エラーメッセージのキーワードで Grep し、エラーが発生しているファイル・行を特定
- **コールパスの追跡**: エラー箇所から呼び出し元をさかのぼり、Controller → Service → Repository などのパスを記録
- **データ・状態の確認**: 関連するモデル・DB スキーマ・設定値を確認
- **類似コード・テストの確認**: 同様の処理の実装例・テストコードを調査

調査を進めながら `hypothesis_log.md` に仮説と検証結果を随時記録する:

~~~markdown
# Hypothesis Log - <investigation-name>

## 調査済みパス

| # | 仮説 | 確認内容 | 結果 |
|---|------|---------|------|
| 1 | （仮説） | （確認したファイル・内容） | 棄却/採用 |

## 棄却理由
- 仮説1: （なぜ違ったか）
~~~

## ステップ4: アウトプット生成

`investigation/summary.md` を作成する:

~~~markdown
# Investigation Summary - <investigation-name>

## 調査対象
（バグの概要・発生状況）

## 関連ファイル
- `path/to/file`: （役割）

## 調査経路
（辿ったコールパスの概要）

## 不明点・リスク

| # | 内容 | 影響 | 状態 |
|---|------|------|------|
| 1 | （不明点） | 高/中/低 | 未解決/解決済 |
~~~

`root_cause.md` を作成する:

~~~markdown
# Root Cause - <investigation-name>

## 根本原因

（コードを読んで確認した根本原因。ファイル名・行番号・コードスニペットを含める）

## 原因の連鎖

（どのような経緯でバグが起きるか）

## 再現条件

（どんな状況で発生するか）
~~~

`fix_proposal.md` を作成する:

~~~markdown
# Fix Proposal - <investigation-name>

## 修正方針

（どのように修正するか）

## 修正箇所

| ファイル | 修正内容 |
|---------|---------|
| `path/to/file` | （何を変更するか） |

## 修正案（コードスニペット）

```
（修正前 → 修正後のコード例）
```

## 注意点・副作用

（修正によって影響を受ける可能性がある箇所）

## テスト方針

（修正を確認するためのテスト方針）
~~~

## ステップ5: 報告とフィードバック

作成したファイルを提示してユーザーにフィードバックを求める。
「解決できなかった不明点」は明示して確認を取る。
